# 🚀 화성 기지 비상 창고 초고속 암호 해독기 (Mars Mission - Step 8)

## 1. 프로젝트 개요
이 프로젝트는 암호화된 ZIP 파일(`emergency_storage_key.zip`)의 비밀번호를 해독하는 고급 파이썬 프로그램입니다. 
단순한 무차별 대입(Brute-force) 방식을 넘어, **다중 코어(멀티프로세싱)** 와 **ZIP 파일의 구조적 취약점**을 찔러 초고속으로 암호를 찾아내는 엔터프라이즈급 최적화 기법이 적용되어 있습니다.

### 🎯 핵심 최적화 기술
1. **수학적 공간 분할 (Base36):** 일반적인 문자열 생성(`itertools`) 대신 36진법 정수 연산을 사용하여 21억 개의 경우의 수를 코어별로 정확히 분할하고 완벽한 이어하기(Resume)를 지원.
2. **ZipCrypto 취약점 공략:** 파일을 끝까지 압축 해제하지 않고, **단 12바이트 암호화 헤더 중 1바이트만 복호화하여 비교**함으로써 99.6%의 오답을 빛의 속도로 스킵.
3. **IPC 병목 현상 제거:** 공유 메모리(Lock) 접근을 최소화하기 위해 각 워커 프로세스별 로컬 캐싱(Local Caching) 기법 적용.
4. **원자적(Atomic) 체크포인트:** 강제 종료나 정전 시에도 단 1회의 탐색 기록도 유실되지 않는 파일 안전 교체 저장 시스템.

## 2. 코드 흐름 & 핵심 알고리즘 완벽 해부

### STEP 1 — 문자열이 아닌 '숫자(N진법)'로 탐색 공간 지배하기
일반적인 크래커는 `aaaaaa`, `aaaaab` 처럼 문자열을 차례대로 생성합니다. 하지만 이 프로그램은 **암호를 36진법 숫자(0 ~ 2,176,782,335)로 취급**합니다.
* **정확한 작업 분배:** 전체 약 21억 개의 탐색 공간을 사용 가능한 멀티코어 CPU 수만큼 정확히 나누어 할당할 수 있습니다.
* **완벽한 이어하기:** 중단되었을 때 '현재 문자열'이 아닌 '진행된 인덱스 숫자'만 저장하므로 구조가 압도적으로 빠르고 효율적입니다.

### STEP 2 — 파이썬의 한계를 넘는 초고속 암호 생성기 (클로저)
N진법 변환 시 매번 반복문 안에서 거듭제곱 연산을 하면 심각한 병목이 발생합니다. 이를 해결하기 위해 각 자리의 가중치(`Base^N`)를 리스트로 미리 계산해 캐싱해 두고, 인덱스(`idx`)만 받아 즉시 바이트(bytes)로 변환해 주는 **전용 클로저(Closure) 함수**를 동적으로 생성하여 무한 반복에 투입합니다.

### STEP 3 — 진정한 비밀무기: ZipCrypto 1바이트 헤더 취약점 공격
전체 파일을 무식하게 압축 해제(Brute-force DEFLATE)하는 것보다 이 방식이 수백 배 빠른 이유는 **ZIP 파일의 레거시 암호화(ZipCrypto) 설계 자체에 있는 구조적 결함**을 찌르기 때문입니다.

**[ZipCrypto의 구조적 결함이란?]**
ZipCrypto는 데이터를 암호화할 때, 실제 압축된 데이터 맨 앞에 **12바이트의 무작위 데이터(Salt)**를 추가합니다. 설계자들은 암호가 맞는지 빠르게 확인할 수 있도록, 12바이트 중 **마지막 12번째 바이트를 무작위가 아닌 ZIP 파일의 메타데이터(평문) 중 일부를 복사**해 넣었습니다.
* 기본적으로 원본 파일의 **CRC32 해시값의 최상위 1바이트**가 들어갑니다.
* 데이터 디스크립터 사용 시 파일 **수정 시간(Time)의 상위 1바이트**가 들어갑니다.
이 CRC32나 수정 시간은 ZIP 파일 헤더에 암호화되지 않은 채 '평문'으로 기록되어 있다는 것이 치명적인 취약점입니다.

**[초고속 해독(Cracking) 시나리오]**
1. **정답지(check_byte) 훔쳐보기:** 파일을 열자마자 평문으로 노출된 메타데이터에서 마지막 12번째 바이트에 들어있어야 할 정답(1바이트)을 미리 계산해서 빼옵니다.
2. **암호화된 12바이트만 떼어오기:** 수십 MB의 ZIP 파일 본문은 무시하고, 오직 데이터 시작점의 **12바이트 암호화 헤더**만 메모리로 읽어옵니다.
3. **1바이트 초고속 검증:** 무작위로 생성한 암호 후보를 가지고 **딱 12바이트만 복호화**를 진행한 후, 마지막 바이트가 훔쳐본 `check_byte`와 일치하는지 확인합니다.

**[놀라운 수학적 효과]**
* 1바이트는 0~255까지 256개의 값을 가집니다. 즉, 완전히 틀린 암호인데도 우연히 마지막 1바이트가 정답과 일치할 확률은 **1/256 (약 0.39%)** 입니다.
* 결과적으로 전체 경우의 수 중 **99.61%의 오답은 수십 바이트의 메모리 연산만으로 마이크로초 단위로 즉시 탈락**합니다. (21억 번의 무거운 압축 해제 연산이 약 820만 번으로 대폭 줄어듭니다.)

### STEP 4 — 멀티프로세싱 병목 현상 제거 (로컬 캐싱과 Lock 최적화)
여러 개의 코어가 협력할 때 가장 큰 적은 **'공유 자원 동기화(IPC)로 인한 병목'**입니다. 

**[최악의 시나리오: 매번 동기화할 경우]**
만약 8개의 코어가 암호를 1번 시도할 때마다 메인 카운터(공유 메모리)를 1씩 올린다면, 8개의 코어는 하나의 카운터에 접근하기 위해 매번 줄을 서서 대기해야 합니다(Lock Contention). 결과적으로 병렬 처리의 이점이 사라지고 잦은 컨텍스트 스위칭으로 인해 싱글 코어보다 속도가 느려지는 참사가 발생합니다.

**[초고속 해결책: 로컬 캐싱(Local Caching)]**
이 코드의 워커 프로세스는 메인 카운터에 매번 보고하지 않습니다.
1. 각 코어는 자신의 전용 메모리(로컬 변수)에 시도 횟수를 기록하며 100,000번(`SYNC_INTERVAL`)의 탐색을 쉬지 않고 전속력으로 달립니다.
2. 딱 **10만 번이 채워지는 순간, 단 1번만 Lock을 걸고 메인 카운터에 값을 한꺼번에 더합니다.**

**[놀라운 성능 향상]**
10만 번의 검증을 할 때 10만 번 발생해야 할 프로세스 간 통신(IPC)과 Lock 대기 시간이 **단 1번**으로 줄어듭니다. 동기화 오버헤드가 **99.999% 감소**하며, 모든 코어가 대기 시간 없이 100%의 CPU 점유율을 유지하며 병렬 처리의 진가를 발휘하게 됩니다.

### STEP 5 — 원자적(Atomic) 체크포인트 저장
진행 상황을 파일에 저장하다가 전원이 나가서 체크포인트 파일이 0kb로 손상되는 불상사를 원천 차단합니다. 
데이터를 `.tmp` 임시 파일에 완벽하게 쓴 뒤, OS 버퍼를 무시하고 디스크에 물리적 기록을 강제(`os.fsync`)한 후에만 `os.replace`를 통해 원본 파일명을 덮어씌웁니다.

## 3. 실행 방법 (CLI 옵션)

다양한 명령행 인자(Arguments)를 통해 탐색 전략을 동적으로 제어할 수 있습니다.

```bash
# 기본 실행 (전체 코어 자동 할당 및 처음부터 탐색)
python door_hacking.py

# 이어하기 (중단된 체크포인트 지점부터 이어서 해독)
python door_hacking.py --resume

# 역순 탐색 (zzzzzz부터 반대로 탐색)
python door_hacking.py --reverse

# 특정 구간 탐색 (특정 문자열 범위 내에서만 탐색)
python door_hacking.py --start aaaaaa --end bzzzzz

# 코어 수 수동 지정 (예: 4코어만 사용)
python door_hacking.py --workers 4
```
